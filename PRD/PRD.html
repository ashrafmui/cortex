PRD: Cortex — AI-Powered Active Learning Layer

Author: Muhaiminul Islam
Version: 1.0
Date: April 16, 2026
Status: Draft
Timeline: 2 weeks (April 21 – May 2, 2026)

1. Problem Statement

Current LLM chat interfaces (ChatGPT, Claude.ai, Gemini) are stateless Q&A pipes. A user asks, the model answers, and the interaction ends. This is a fundamentally broken model for learning:

  • No retention loop. The user reads a response and forgets 80% within 48 hours (Ebbinghaus curve). There is no mechanism for spaced recall.
  • Passive consumption. Chat encourages reading, not doing. The learner never has to reconstruct knowledge — they just receive it.
  • No structure. There's no concept map, no dependency graph, no sense of "what you know" vs. "what you don't." The user is responsible for all metacognition.
  • Flat difficulty. The model doesn't calibrate to the learner's actual understanding. A PhD and a freshman get the same response to the same prompt.

These are solved problems in learning science. Spaced repetition, active recall, Socratic questioning, mastery-based progression, and interleaving have decades of evidence behind them. No product currently bridges the gap between LLM capability and pedagogical rigor.

Cortex is that bridge.

2. Product Vision

An AI learning layer that wraps any LLM API (Claude as primary, model-agnostic by design) and transforms passive chat into structured, adaptive, recall-driven learning sessions. The user doesn't "chat with AI" — they learn through AI, with the system orchestrating the pedagogy invisibly.

Core Thesis

The LLM is the knowledge engine. Cortex is the learning engine. Separation of concerns.

The LLM generates content, explanations, analogies, and assessments. Cortex decides when to explain, when to quiz, when to revisit, what difficulty to use, and how to connect new knowledge to existing knowledge — all driven by a learner model that updates in real time.

3. Target User

Primary: Self-directed learners studying technical subjects (programming, embedded systems, math, EE, physics, system design). Adults who know what they want to learn but lack structure and retention.

Secondary: Students supplementing coursework. Interview preppers. Career-switchers building new domain knowledge.

Anti-user: Someone who wants a quick factual answer. That's what vanilla Claude is for. Cortex is for when you want to own the knowledge.

4. Core Feature Set (MVP — 2-Week Scope)

4.1 Learning Sessions (not chats)

The fundamental interaction model is a session, not a conversation. A session has:

  • A declared topic or learning goal (user-provided or AI-suggested)
  • A mode that the system manages dynamically:
    • TEACH — The AI explains a concept. Used when the learner model shows no prior exposure.
    • QUIZ — The AI poses a question and evaluates the learner's answer. Used to test retention.
    • SOCRATIC — The AI asks leading questions to guide the learner toward a conclusion. Used for deeper understanding.
    • REVIEW — The AI resurfaces previously-learned concepts due for spaced repetition.
  • A difficulty tier (foundation → working → deep → applied) that adapts based on quiz performance.

The user sees a clean interface. They don't see mode labels or difficulty tiers — the system manages transitions silently. The experience feels like a great tutor: sometimes it explains, sometimes it asks, sometimes it pushes.

4.2 Learner Model (Knowledge Graph)

A lightweight per-user knowledge representation:

ConceptNode {
  id: string
  topic: string              // e.g., "Pointers in C"
  parent_topic: string       // e.g., "Memory Management"
  mastery: float [0.0–1.0]   // computed from quiz performance + decay
  last_reviewed: timestamp
  review_interval: duration   // SM-2 algorithm interval
  exposure_count: int
  difficulty_tier: enum
}
  • Mastery decays over time using a simplified SM-2 spaced repetition curve.
  • Mastery increases when the user answers quiz questions correctly.
  • Mastery decreases on incorrect answers, scaling by confidence (if the user was confident and wrong, mastery drops more).
  • The graph is built incrementally — Cortex extracts concepts from every session and upserts nodes.

This is the product's moat. Every session makes the system smarter about this specific learner.

4.3 Session Orchestrator (The Brain)

A middleware layer (not the LLM — deterministic logic) that decides:

  1. What to teach next. Based on the knowledge graph: prioritize low-mastery prerequisites, then introduce new concepts, then review due items.
  2. What mode to use. New concept → TEACH. Seen before, mastery < 0.4 → SOCRATIC. Mastery 0.4–0.7 → QUIZ. Due for review → REVIEW.
  3. What difficulty to set. Maps directly from the concept's difficulty_tier.
  4. When to transition. After 2–3 exchanges in TEACH mode, shift to QUIZ. After a successful QUIZ, advance difficulty or move to the next concept.

The orchestrator constructs the LLM prompt dynamically. The user never writes a system prompt — Cortex builds it.

4.4 Prompt Engineering Layer

The orchestrator outputs structured system prompts to the LLM. Examples:

TEACH mode prompt (simplified):

You are tutoring a learner on [TOPIC]. Their current mastery level is [TIER].
Explain [CONCEPT] at the [TIER] level. Use concrete examples.
Do not cover [PREREQUISITE CONCEPTS] — the learner already knows these.
After explaining, ask one comprehension check question.
Respond in JSON: { "explanation": "...", "check_question": "...", "concepts_introduced": [...] }

QUIZ mode prompt (simplified):

Generate a [DIFFICULTY] question testing [CONCEPT].
The learner has seen this concept [N] times and last scored [X].
Include a grading rubric. After the learner answers, score 0.0–1.0 with reasoning.
Respond in JSON: { "question": "...", "rubric": "...", "hints": [...] }

All LLM responses are structured JSON. The UI renders them. The LLM never directly controls the UI — it provides data, Cortex renders it. This keeps the experience predictable and debuggable.

4.5 Dashboard

  • Topic map: Visual representation of the knowledge graph. Nodes colored by mastery (red → yellow → green). Clickable to start a session on that topic.
  • Review queue: Concepts due for review today, ranked by urgency.
  • Session history: Past sessions with timestamps and mastery deltas.
  • Streak / stats: Daily streak, total concepts learned, average mastery.

4.6 Out of Scope (v1)

  • Multi-user / collaborative learning
  • Mobile app (responsive web only)
  • Offline mode
  • Image/diagram generation within sessions
  • PDF/document ingestion as learning material (future: huge feature)
  • LTI integration (LMS platforms)
  • Voice interaction

5.2 Data Model (Prisma Schema — Core Tables)

model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  createdAt DateTime @default(now())
  sessions  Session[]
  concepts  ConceptNode[]
}

model ConceptNode {
  id              String   @id @default(cuid())
  userId          String
  user            User     @relation(fields: [userId], references: [id])
  topic           String
  parentTopic     String?
  mastery         Float    @default(0.0)
  difficultyTier  String   @default("foundation") // foundation | working | deep | applied
  lastReviewed    DateTime?
  reviewInterval  Int      @default(1) // days, SM-2 interval
  easeFactor      Float    @default(2.5) // SM-2 ease factor
  exposureCount   Int      @default(0)
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
  quizResults     QuizResult[]

  @@unique([userId, topic])
  @@index([userId, mastery])
  @@index([userId, lastReviewed])
}

model Session {
  id          String   @id @default(cuid())
  userId      String
  user        User     @relation(fields: [userId], references: [id])
  goal        String
  mode        String   @default("TEACH") // TEACH | QUIZ | SOCRATIC | REVIEW
  messages    Json     // Array of { role, content, mode, timestamp }
  conceptsHit String[] // concept IDs touched in this session
  startedAt   DateTime @default(now())
  endedAt     DateTime?
}

model QuizResult {
  id          String   @id @default(cuid())
  conceptId   String
  concept     ConceptNode @relation(fields: [conceptId], references: [id])
  score       Float    // 0.0 - 1.0
  confidence  String?  // "low" | "medium" | "high" (self-reported)
  question    String
  answer      String
  feedback    String
  createdAt   DateTime @default(now())
}

6. Key Technical Decisions & Tradeoffs

LLM-as-data-source, not LLM-as-controller

The LLM never decides what happens next. It generates content within constraints the orchestrator sets. This is a deliberate architectural decision:

  • Predictability. Deterministic orchestration means the learning flow is testable and debuggable.
  • Cost control. The orchestrator can batch, cache, and skip LLM calls when it has enough data.
  • Portability. Swap Claude for GPT-4o or Gemini without changing any learning logic.

Structured JSON output over free-text

Every LLM response is parsed JSON. The UI renders structured data, not raw markdown. Benefits: consistent UX, extractable concepts/scores, no prompt injection into the UI. Cost: slightly more complex prompts, occasional parse failures (mitigate with retry + fallback).

SM-2 over more complex SRS algorithms

SM-2 (SuperMemo 2) is the simplest well-validated spaced repetition algorithm. FSRS and SM-18 are more accurate but add complexity without meaningful MVP benefit. SM-2 is ~50 lines of code and well-understood. Upgrade path is clear if needed.

Monorepo over microservices

At this scale (single developer, 2-week build, <1000 users at launch), a monorepo Next.js app with API routes is the right call. Microservices would add deployment complexity, network latency, and debugging overhead with zero benefit.

PostgreSQL over SQLite

SQLite would work for a single-user prototype but doesn't support concurrent connections in serverless (Vercel). PostgreSQL via Supabase/Neon gives us a managed database with connection pooling, free tier, and a migration path to production.

10. Future Roadmap (Post-MVP)

These are explicitly out of scope for the 2-week build but represent the natural evolution:

  1. Document ingestion — Upload a PDF/textbook chapter, Cortex extracts concepts and builds a learning path from it. Huge feature.
  2. Learning paths — Curated or AI-generated multi-session curricula (e.g., "Learn Embedded C in 30 days").
  3. Code sandbox — For programming topics, embed a live code editor where the learner writes and runs code as part of QUIZ/APPLIED mode.
  4. Multiplayer — Study groups, shared knowledge graphs, peer quizzing.
  5. Concept prerequisites — Explicit dependency edges in the knowledge graph. "You can't learn X until you know Y."
  6. Export — Anki deck export from the knowledge graph. Markdown notes export.
  7. Voice mode — Conversational learning via speech-to-text + TTS.
  8. Analytics — Forgetting curve visualization, optimal study time recommendations.

11. Open Questions

  1. Concept granularity. How fine-grained should concept nodes be? "Pointers" vs. "Pointer arithmetic" vs. "Pointer to pointer." Leaning toward letting the LLM decide based on topic scope, then merging duplicates with fuzzy matching.
  2. Confidence self-report. Should the user rate their confidence before answering a quiz? Adds a data point for mastery calculation but adds friction. Leaning toward including it as an optional one-tap UI element.
  3. Session length control. Fixed length (e.g., 10 questions) vs. adaptive (stop when mastery hits a threshold)? Leaning toward adaptive with a soft cap at 15 minutes.
  4. Multi-model support priority. How soon do we need OpenAI/Gemini support? The Vercel AI SDK makes this low-effort, but prompt tuning per model is non-trivial. Leaning toward Claude-only for MVP, abstract later.
